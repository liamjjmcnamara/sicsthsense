Detailed Description

This class is provided to you to make working with multiple sensors simple. It can be used to scan for new sensors, add them to the system, connect to them at any time, receive data from them, disconnect, and also remove them from the system.

Getting started:

  1. Create a new class of your own that extends this class. For example: 
    public class MyCustomSensorsManager extends SensorsManager
  2. Create a new sensor class that extends extends this class. For example: 
    public class MyHeartRateSensor extends SensorBase
    See SensorBase for a full explanation of how to write a new sensor class.
  3. Define Integer codes in your class for each type of sensor you will use in your system. For example: 
    public static final int SENSOR_TYPE_MY_BLUETOOTH_HEART_RATE_SENSOR = 1;
  4. Define String codes in your class for each variable that these sensors will provide. For example: 
    public static final String DATA_TYPE_HEART_RATE = "hrt";
  5. If your system will contain external (Bluetooth) sensors, you wil need to call SensorsManager#addDeviceNameCode(String, Integer) in your new class's constructor for each type of external sensor you wish to use. For example: 
    addDeviceNameCode("HR", SENSOR_TYPE_MY_BLUETOOTH_HEART_RATE_SENSOR);
  6. Finally, you will need to override the method SensorsManager#instantiateSensor(Integer, String). It is best to think of this method as a "Sensor factory", which creates new sensor objects based on the sensor type provided. For example: 
            protected SensorBase instantiateSensor(Integer sensorType, String sensorAddress) {
            SensorBase sensor = null;
            switch(sensorType){
            case SENSOR_TYPE_MY_BLUETOOTH_HEART_RATE_SENSOR:
                sensor = new MyHeartRateSensor();
                break;
            default:
                break;
            }       
            return sensor;
        }

Once these initial steps have been completed for each sensor and variable you want to use in your system, you are ready to use your new MyCustomSensorsManager class in your project. You can get a create an instance of your manager as follows: 

MyCustomSensorsManager mySensorsManager = new MyCustomSensorsManager(context);

Now you may use this object to perform any sensor administration work, from scanning for sensors, to receiving data from them.

void se.sics.sensorservices.SensorsManager.addDeviceNameCode ( String  deviceNameCode,
Integer  sensorType 
)

Adds a Bluetooth name code to the list of available name codes. A name code is a two-character string that is used to identify a Bluetooth sensor during a scan, and should always be the first two characters of your Bluetooth sensor's visible name. For example, if you have a Bluetooth heart rate sensor, you should decide on a code (e.g. HR) and place this at the beginning of the sensor's Bluetooth name, e.g. ("HR 1234"). This code will then be used when scanning for Bluetooth sensors, and any devices with names beginning with "HR" will be assumed to be one of your heart rate sensors. For this reason, it is strongly recommened to use a code that is not likely to be found in other devices' names.

Parameters
deviceNameCodeA two-character string that will identify your Bluetooth sensor
sensorTypeThe type of sensor that this code will refer to. You should define Integer codes for each of your sensors somewhere (a good place to do this is in your custom implementation of the SensorsManager class).

Connects to all available sensors, and optionally start logging data to file. Since some sensors may perform their connection routines asynchronously, it is not guaranteed that all sensors will be connected once this method has comepleted. Instead, you should provide this SensorsManager with a handler using SensorsManager#setHandler(Handler), and listen for a the message with the type SensorsManager#MESSAGE_TYPE_CONNECTION_COMPLETE.

Parameters

HashMap<String, Object> se.sics.sensorservices.SensorsManager.getSummaryData ( )

Returns a set of all available summary data (from all sensors). Summary data are values that sensors may have calculated during the track that are not suitable for periodic logging (a good example is that whilst current heart rate should be logged periodically, average heart rate over the entire track does not need to be logged periodically, and is more suited to just being reported on request). It is up to each sensor to decide what summary data it returns, and a sensor does not have to return any summary data.

Returns
A HashMap of summary data values (stored as Objects), indexed by their type code. This HashMap may be empty (initialised but 0 size).

void se.sics.sensorservices.SensorsManager.scanForNewSensors ( ArrayList< Integer >  desiredSensorTypes,
boolean  automaticallyAdd 
)

Starts a scan for any sensors that are not currently stored in the persistent sensor configuration. To be informed of any new sensors found, create a SensorScanListener in your code and pass it as a parameter when calling this method. When a new sensor is found, the method SensorScanListener#newSensorFound(SensorBase, boolean) will be called in your listener, with the new sensor provided as a parameter.

Parameters
sensorScanListenerA SensorScanListener that will be notified of any events that occur during the scan (e.g. new sensor detected)
desiredSensorTypesAn ArrayList of sensor type codes. If this is not null, only sensors matching the codes in this list will be considered
automaticallyAddIf true, any sensors found will be automatically added to the configuration. If false, it is up to your SensorScanListener to decide what to do. Note that you may still use a SensorScanListener to receive events, even if this is set to true.

void se.sics.sensorservices.SensorsManager.startRecordingFromSensors ( )

Signals to each connected sensor that it should begin recording and / or transmitting data. If there is at least one sensor connected, this will cause data messages to be sent to the handler provided at regular intervals. These messages are of the type SensorsManager#MESSAGE_TYPE_DATA, and the sending interval can be set by calling SensorsManager#setDataMessageInterval(int). If this is not called, the interval defaults to SensorsManager#FREQUENCY_1HZ_MS

void se.sics.sensorservices.SensorsManager.stopRecordingFromSensors ( )

Signals to each connected sensor that it should stop recording and / or transmitting data. Once you call this method, you will not receive any more sensor data. Additionally, when this method is called, each sensor may optionally perform some final processing (such as generating a summary report of the data it has collected). If you want your sensor to do something when this method is called, you should override SensorBase#stopRecording() and place your code in your overriding method.
